# metaopt-proposal-rollover

## Overview

Leaf worker skill invoked by the `ml-metaoptimization` orchestrator during `ROLL_ITERATION`.
Its single responsibility is to filter and curate the `next_proposals` pool so it becomes a clean `current_proposals` pool for the next iteration.

This skill does **not** generate new proposals. It only carries over, discards, or merges existing ones based on the latest experiment results, key learnings, and proposal invalidations from the results analysis phase.

The orchestrator dispatches this skill as a `strong_reasoner` subagent. Prefer a strong reasoning model (e.g. Opus 4.6 fast); fall back to a capable general model (e.g. GPT-5.4).

## Input Contract

The orchestrator provides all inputs via the subagent prompt.

### Standard Envelope (provided by orchestrator — inline dispatch)

| Field | Type | Description |
|-------|------|-------------|
| `campaign_id` | string | Campaign identifier |
| `current_iteration` | integer | Current iteration number |

Note: `slot_id` and `attempt` are omitted for inline dispatch (rollover does not consume a slot).

### State-Specific Inputs

| Input | Type | Description |
|-------|------|-------------|
| `next_proposals` | list of proposal objects | Proposals accumulated during the current iteration (generated by ideation workers and buffered while the batch ran) |
| `analysis_summary` | object | Persisted analysis output from `state.selected_experiment.analysis_summary` containing `judgment`, `new_aggregate`, `delta`, `learnings`, `invalidations`, and `carry_over_candidates` |
| `key_learnings` | list of strings | Cumulative learnings across all iterations |
| `completed_experiments` | list of experiment records | History of all experiments run so far, including the one just completed |
| campaign goal | object | Normalized objective fields: `metric`, `direction`, `aggregation_method`, `aggregation_weights`, `improvement_threshold` |
| `proposal_policy` | object | Contains `current_target`, `current_floor`, `next_cap`, and `distinctness_rule` |
| stop conditions progress | object | `iterations_completed`, `no_improve_iterations`, `max_iterations`, `max_no_improve_iterations` |

### Proposal Record Shape

Each proposal in `next_proposals` is a full proposal record (see `ml-metaoptimization/references/contracts.md` Proposal Record Shape):

**Orchestrator-owned fields:**
- `proposal_id`: non-empty string, unique within the campaign
- `source_slot_id`: non-empty string identifying the originating slot
- `creation_iteration`: positive integer
- `created_at`: ISO 8601 timestamp

**Worker-provided fields:**
- `title`: concise name (≤ 12 words)
- `rationale`: why this change is expected to improve the metric
- `expected_impact`: object with `direction` and `magnitude`
- `target_area`: which pipeline area this targets

Carry-over proposals must preserve ALL fields unchanged.

## Output Contract

Return a structured response containing:

### 1. Filtered proposal set

A list where each entry from `next_proposals` is annotated with a disposition:

| Disposition | Fields |
|-------------|--------|
| `carry_over` | Original proposal unchanged, retaining all metadata |
| `discard` | Proposal identifier + `reason` (e.g. "invalidated by results analysis", "superseded by learning: <X>", "duplicate of carry-over proposal <Y>") |
| `merge` | List of source `proposal_id` values + merged candidate (same shape as an ideation candidate: `title`, `rationale`, `expected_impact`, `target_area`) + `merge_rationale`. The orchestrator enriches the merged candidate with `proposal_id`, `source_slot_id` (set to `"rollover"`), `creation_iteration`, and `created_at` before persisting. |

### 2. Summary statistics

```
carried_over: <int>
discarded:    <int>
merged:       <int>  (count of merge groups, not source proposals)
final_pool_size: <int>
```

### 3. Pool health flag

If `final_pool_size` < `proposal_policy.current_floor`, include:

```
needs_fresh_ideation: true
shortfall: <current_floor - final_pool_size>
```

This tells the orchestrator that the next iteration must prioritize aggressive ideation to refill the pool before selection can proceed.

## Filtering Rules

Apply these rules in order:

### Step 1 — Invalidation sweep

Scan the results analysis output for explicit proposal invalidations. Any proposal that was explicitly invalidated by the analysis must be discarded unless there is strong contrary evidence in `key_learnings` that rehabilitates it. Document the evidence if overriding an invalidation.

### Step 2 — Duplicate and overlap detection

Compare all remaining proposals pairwise. Two proposals overlap if they would produce materially the same code change or test the same hypothesis via the same mechanism. Apply the campaign's `distinctness_rule`:

- `non_overlapping` (default): discard the weaker duplicate (prefer the one with more specific rationale or the one generated more recently)
- If both are equally strong, merge them into a single refined proposal

### Step 3 — Superseded-by-learning check

For each remaining proposal, check whether any entry in `key_learnings` renders it moot. A proposal is superseded when:
- A learning proves the hypothesis wrong
- A learning shows the approach was already tried and failed
- A learning reveals that the proposal's target code path no longer exists or has been refactored

Discard superseded proposals with a reason referencing the specific learning.

### Step 4 — Merge pass

Identify proposals that are complementary — they address different aspects of the same improvement area and would be stronger combined. Merge them into a single proposal that:
- Combines the rationales
- Preserves all source identifiers for traceability
- Returns the merged result as a candidate (worker-provided fields only: `title`, `rationale`, `expected_impact`, `target_area`); the orchestrator assigns a new `proposal_id` and sets `source_slot_id = "rollover"`
- Remains distinct from other carry-over proposals (re-check `distinctness_rule`)

### Step 5 — Pool size check

Count the final carry-over + merged proposals. If below `current_floor`, set `needs_fresh_ideation: true`.

## Behavioral Rules

1. **Do not generate new proposals.** This skill only filters, merges, or discards. New ideation happens in `MAINTAIN_BACKGROUND_POOL` via ideation workers.

2. **Respect invalidations.** If the results analysis explicitly invalidated a proposal, discard it. Override only with documented contrary evidence.

3. **Respect the distinctness rule.** After merging, verify the merged proposal does not overlap with any other carry-over. If it does, merge further or discard the weaker one.

4. **Preserve proposal identity.** Carry-over proposals must retain their original identifier, metadata, and creation context. Do not rewrite or paraphrase them.

5. **Be conservative.** When uncertain whether to discard or carry over, carry over. The next iteration's selection process will rank proposals anyway.

6. **Respect pool bounds.** The output pool should not exceed `next_cap`. In practice it rarely will (since inputs come from a capped `next_proposals`), but enforce the ceiling if needed by discarding the weakest proposals.

7. **Flag ideation shortfall.** If the filtered pool is below `current_floor`, the orchestrator needs to know immediately so it can prioritize ideation in the next iteration.

8. **No side effects.** This skill does not write to state files, modify the working tree, or interact with the queue backend. It returns a structured response; the orchestrator handles persistence.

## Common Mistakes

| Mistake | Fix |
|---------|-----|
| Generating new proposals during rollover | Only filter, merge, or discard — ideation is a separate lane |
| Ignoring explicit invalidations from results analysis | Always discard invalidated proposals unless documented contrary evidence exists |
| Merging proposals that violate `distinctness_rule` | Re-check the merged result against all other carry-overs for overlap |
| Stripping original metadata from carry-over proposals | Preserve identifiers, rationale, source slot, and creation iteration |
| Discarding uncertain proposals aggressively | When in doubt, carry over — selection will rank them later |
| Not flagging pool shortfall | If below `current_floor`, set `needs_fresh_ideation: true` so the orchestrator can act |
| Rewriting proposal text on carry-over | Carry-over means unchanged — do not paraphrase or "improve" the wording |
| Counting merged source proposals instead of merge groups | `merged` count = number of merge operations, not number of source proposals consumed |
